# 余光阅读器 MVP：技术选型与实施清单

> 依据：`需求文档.txt`  
> 目标：先完成一个稳定、流畅、可日常使用的 Windows 本地阅读器；只做 MVP 必需能力，不预埋云同步、插件系统等架构。

## 1. 需求结论与边界

### MVP 必须做到

- 支持打开本地 TXT、EPUB、MOBI，章节之间连续向下阅读。
- 老板键全局生效；隐藏无动画、无提示、无任务栏残留，恢复时不抢原应用焦点。
- 鼠标滚轮和触控板滚动平滑；自动滚动按实际时间推进，兼容 60Hz/120Hz。
- 按“章节 + 段落 + 段内偏移”恢复进度，窗口尺寸或字体变化后仍回到同一段文字。
- 大文件分块/分章加载，DOM 中只保留当前位置附近的章节。
- 提供字体、字号、行距、透明度、置顶、鼠标穿透、快捷键等基础设置。
- 托盘常驻；工具栏、边框和滚动条在鼠标移出后自动隐藏。

### 需求冲突的处理结论

1. 文档对默认键存在冲突。当前同时保留 **Ctrl+Tab** 和 **F8**：Ctrl+Tab 通过 `RegisterHotKey` 接入，F8 作为兼容回退；Ctrl+Tab 与浏览器/编辑器页签切换的兼容性仍需人工验收。
2. MVP 不引入 `WH_KEYBOARD_LL` 和输入模拟；老板键采用两键 Ctrl+Tab，若系统拒绝注册则明确保留 F8，不拦截普通 Alt 组合键。
3. “不抢焦点”和“键盘控制正文”不能同时无条件成立。约定如下：
   - 显示、恢复、自动滚动和鼠标滚动不得抢走当前办公软件焦点；
   - 老板键和 F7 是全局键；
   - 方向键、空格、Page Up/Down 只在阅读器交互模式下处理，不全局拦截办公软件按键；
   - 鼠标穿透开启后不再接收鼠标滚动/拖动，用户通过托盘关闭穿透。
4. MVP 的 EPUB/MOBI 支持范围收敛为正文、标题、章节顺序、段落和常见行内样式；本地图片延后。MOBI 仅支持无 DRM 文件，并在真实样本验证通过前标记为 Beta；不做 DRM 破解、脚注弹窗、批注、目录编辑、音视频和复杂脚本内容。

### 本次复盘后的取舍

- 主线使用标准 WPF `WebView2`；`WebView2CompositionControl` 不再作为 MVP 前置条件。首版透明度优先使用整窗 `Opacity`，组合控件仅在标准控件出现无法规避的遮挡问题时重新评估。
- 不采纳“默认改成普通任务栏窗口、再拆主窗口/阅读窗口/悬浮模式”的建议：它与余光阅读器的核心定位冲突，也会同时维护多套窗口状态。保留无边框浮窗，但必须补齐拖动、边缘缩放和位置记忆。
- 先完成“一本 TXT 从打开到连续阅读、隐藏恢复、精确续读”的纵向闭环，再扩充 EPUB/MOBI 样本覆盖；避免三种格式、设置页和滚动引擎同时推进。
- 触控板优先保留浏览器原生高精度滚动；自定义惯性只处理传统鼠标滚轮，避免双重平滑。
- “已实现”与“已验收”分开：只有自动测试或人工验收通过的条目才勾选；已有代码但功能不完整的条目保持未勾选并注明现状。
- 架构只做两处必要拆分：`ReaderSession/ReaderBridge` 管阅读会话与 WebView2，`WindowBehavior` 管热键/隐藏/穿透。暂不引入 DI 容器、全套接口、MVVM 框架、Serilog 或多窗口体系。

### `new.txt` 界面复盘结论（2026-07-15）

- 接受并前置处理四个已暴露问题：顶部白边、按钮/文字相互遮挡、阅读区层次弱、窗口缩放后布局拥挤；在这些问题收口前暂停继续堆设置和格式能力。
- 标准 WPF `WebView2` 存在 airspace：叠在控件上方的 WPF 按钮曾出现“看得见但点不到”。MVP 只保留一套正常操作界面，放在 `reader.html` 的布局流中；WPF 负责窗口、系统能力和页面未加载时的错误兜底，不再把可点击控件悬浮覆盖在 WebView2 上。
- 吸收“紧凑顶部工具栏 + 可折叠目录 + 居中正文 + 低存在感提示”的结构，但它只在交互时出现，不能变成永久占据上下空间的标准三栏窗口。MVP 工具栏仍只放打开、目录、自动滚动、速度/排版入口、穿透和设置；`new.txt` 提到的最近阅读、搜索、主题、窗口模式、右侧辅助区暂不进入 MVP。
- WPF 根背景、WebView2 默认背景及 `html/body` 使用同一不透明暖白色；先消除透明叠层、外层 `Padding` 和未验证圆角造成的露底，再考虑轻阴影和圆角，不为视觉效果重新引入 CompositionControl。
- 用少量 CSS 变量统一颜色、间距、圆角和字号，不建设组件库或前端构建链。正文行宽控制在约 `38em～42em`，正文默认 `18px～20px`、行高约 `1.8`。
- 依用户反馈继续隐藏正文和目录的原生滚动条，但必须保留滚轮、触控板、拖动和键盘滚动能力；不改成始终可见的细滚动条。
- 响应式验收至少覆盖 `720×520`、`560×420`、`360×240`，并在 Windows 缩放 `100%/150%/200%` 下检查白边、遮挡、目录折叠和正文可读性。

## 2. 技术选型

| 领域 | 选择 | 原因 |
|---|---|---|
| 平台 | Windows 桌面，优先 x64 | 老板键、托盘、置顶、焦点和窗口样式均依赖 Win32 |
| 运行时 | **C# + .NET 10 LTS** | 新项目不再选临近结束支持的 .NET 8；.NET 10 支持到 2028-11-14，[官方支持周期](https://dotnet.microsoft.com/en-us/platform/support/policy) |
| 窗口 UI | WPF | 与需求建议一致，成熟、依赖少，便于调用 Win32 |
| 阅读界面归属 | 单一 `reader.html/css/js` 界面；WPF 只管窗口和系统能力 | 避免标准 WebView2 airspace 导致 WPF 悬浮按钮被盖住，也避免维护两套工具栏与视觉规则 |
| 正文 | `Microsoft.Web.WebView2` 的标准 WPF `WebView2` | 已验证自包含启动；airspace 用“控件进入 HTML 布局流”规避，不为可规避的覆盖问题引入 CompositionControl 依赖，[WPF 官方说明](https://learn.microsoft.com/en-us/microsoft-edge/webview2/platforms/wpf) |
| 滚动实现 | 原生 JavaScript + CSS，单个 `requestAnimationFrame` 循环 | 以帧间隔计算位移，统一处理滚轮缓动、拖动和自动滚动，不引入前端框架 |
| EPUB | `VersOne.Epub` `3.3.3`（锁定版本） | 避免自行实现 EPUB 容器、清单和阅读顺序解析，[项目文档](https://os.vers.one/EpubReader/) |
| MOBI | `Roler.Toolkit.File.Mobi` `1.0.5`（MIT） | 纯托管 .NET 解析，直接读取无 DRM MOBI 文本；先用统一章节切分，避免引入 Calibre/native 子进程，[项目源码与用法](https://github.com/rolerzhang/RolerFileToolkit) |
| TXT 编码 | .NET 后台读取；BOM/严格 UTF-8 检测，中文文件回退 GB18030 | MVP 允许把文本读入内存，但正文必须分章送入 WebView2；若 500 万字验收出现内存问题再改流式解析 |
| 状态存储 | `System.Text.Json`，写入 `%LocalAppData%\AfterglowReader` | 设置与进度数据量小，JSON 足够；无需 SQLite、EF Core |
| 托盘 | `System.Windows.Forms.NotifyIcon` | 系统自带，避免额外托盘库 |
| 系统能力 | 少量 P/Invoke：窗口扩展样式与 `RegisterHotKey` | 只封装需求用到的 Win32 API；MVP 不使用低级键盘钩子 |
| 测试 | xUnit 单元测试 + 一份人工 Windows 验收清单 | 算法和持久化自动测；焦点、输入和帧率用真实桌面验收更可靠 |
| 发布 | `win-x64` 自包含目录包；WebView2 Evergreen Runtime | 用户无需预装 .NET；安装/启动时检测 WebView2，[官方分发说明](https://learn.microsoft.com/en-us/microsoft-edge/webview2/concepts/distribution) |

### 刻意不采用

- 不使用 Electron、Avalonia、WinUI 3：当前是纯 Windows 项目，WPF 更直接。
- 不使用 React/Vue、Node 构建链：正文页只有排版、滚动和消息通信，原生 JS 足够。
- 不使用数据库、仓储模式、通用事件总线或依赖注入容器。
- 不要求用户安装完整 Calibre，也不自行实现 MOBI 二进制解析；先使用一个锁定版本的纯托管解析包，若真实样本覆盖不足再评估 native fallback。
- 不做常驻多进程架构、插件系统、账号、云同步、在线书库、自动更新。
- 不一次性渲染整本书，也不为未知规模提前实现复杂通用虚拟列表框架。

## 3. 最小代码结构

只建两个项目，避免为“分层”拆出大量程序集：

```text
AfterglowReader.slnx
├─ src/AfterglowReader/          # WPF 主程序
│  ├─ App.xaml
│  ├─ Windows/                   # 阅读窗、设置窗
│  ├─ Books/                     # TXT/EPUB/MOBI 读取和章节模型
│  ├─ Reader/                    # WebView2 外壳、消息和进度锚点
│  ├─ SystemIntegration/         # 热键、窗口样式、托盘
│  ├─ Persistence/               # JSON 设置与进度
│  └─ Assets/Reader/             # reader.html/css/js
└─ tests/AfterglowReader.Tests/  # 纯逻辑单元测试
```

- 不引入完整 MVVM 框架。设置窗使用简单绑定；窗口生命周期和原生消息保留在 code-behind，文件解析、进度、热键等逻辑放到小型服务类。
- C# 与正文页只通过少量 JSON 消息通信：`loadChapters`、`applySettings`、`restoreAnchor`、`progressChanged`、`requestChapter`。
- TXT/EPUB/MOBI 统一为 `Book -> Chapter -> Paragraph`。段落生成稳定 ID，正文页不关心文件格式。

### 最小持久化数据

```text
AppSettings
  windowBounds, opacity, topmost, clickThrough
  fontFamily, fontSize, lineHeight
  wheelSpeed, inertia, autoScrollSpeed, pauseAutoScrollOnHover
  bossKey, autoScrollKey, hideProgressPercent

BookProgress（以规范化文件完整路径为键）
  chapterId
  paragraphId
  paragraphOffsetPx
  overallPercent
  updatedAt
```

- 保存采用“临时文件写完后替换”的原子写入，异常或断电时不留下半份 JSON。
- 不把绝对 `scrollTop` 作为唯一进度；字体、窗口或 DPI 改变时用段落锚点重新定位。

### P0.5：托盘菜单与退出卡死停线项（2026-07-15）

> 状态：**代码修复已接入，人工验收待完成。** 新安装包已经重新生成，但在托盘五项命令的人工矩阵通过前，仍不把托盘菜单标记为最终可用。共同故障链位于 WinForms 托盘回调、WPF Dispatcher、关闭保存和多实例生命周期，不是 WebView2 渲染本身。

#### 已观察到的现象与证据

- 用户反馈托盘中的“显示/恢复、打开书籍、鼠标穿透、设置、退出”都会表现为卡住或无响应；五项的共同入口是 `TrayService` 的 `ContextMenuStrip` 点击回调。
- `TrayService.cs` 当前在 WinForms 菜单 Click 事件栈内直接同步调用 WPF `Action`，没有统一经过 `MainWindow.Dispatcher.BeginInvoke/InvokeAsync`。其中“打开书籍”和“设置”又立即进入模态对话框，菜单事件尚未返回时可能出现菜单不收起、对话框在后台、用户误判为程序卡死。
- `MainWindow.OnClosed` 当前在 WPF UI 线程上对 `SaveProgressNowAsync` 和 `SaveSettingsAsync` 使用 `.GetAwaiter().GetResult()`；底层 `ReaderStateStore.WriteJsonAtomicAsync` 包含异步 `await`。异步续体一旦需要回到 UI 同步上下文，就会与正在同步等待的 UI 线程互相等待。
- 热键释放、托盘 `Dispose`、WebView2 `Dispose` 和其他资源释放均排在同步保存之后；退出保存一旦阻塞，这些清理步骤永远无法执行，进程和托盘图标会残留。
- 应用没有单实例保护。诊断中可以再次启动第二个 `AfterglowReader.exe`；后启动实例日志出现 `CtrlTab=False; F8=False; F7=False`，证明旧实例仍占用全局热键。多个相同托盘图标会分别控制不同窗口，造成“显示/穿透/退出没有效果”的错觉。
- 延迟进度保存采用 fire-and-forget；立即保存只取消旧 CTS，没有等待旧写入实际结束。两个保存任务或两个进程可能同时写同一个 `progress.json.tmp/settings.json.tmp`，存在共享冲突、临时文件被抢先替换和异常关机风险。
- `ShowWithoutActivation` 本身只有 `Show/SetWindowPos`，`ToggleClickThrough` 本身只有窗口扩展样式切换；它们没有长耗时逻辑。若这两项也无响应，应优先判断 UI 线程已经阻塞、菜单回调未正确投递或用户点到了残留实例。
- Windows 应用日志未发现对应的 `Application Hang` 记录；这更符合 UI 线程等待、隐藏模态窗口或错误实例仍存活，而不是 WebView2/CPU 无限循环。

#### 修复实施清单

- [x] 为托盘动作建立唯一 WPF Dispatcher 边界：`TrayService` 不直接操作 WPF 对象；Click/DoubleClick 只投递命令并立即返回，所有命令由主窗口 Dispatcher 按顺序执行。

- [x] “打开书籍”和“设置”必须在菜单 Click 回调返回后再显示；打开期间设置防重入标记，连续点击只保留一次请求。

- [ ] 将设置占位 `MessageBox` 从托盘主流程移除；功能未实现时使用可见、非阻塞的状态提示，不能让用户误以为已经进入设置页。
- [x] 重写退出为单一异步状态机，例如 `RequestShutdownAsync`：只允许进入一次，先标记正在退出并拒绝新菜单命令，再取消开书/延迟保存任务，最后等待保存与资源释放。

- [x] 保存失败时记录明确错误并继续完成退出；不得因为 `progress.json/settings.json` 写入失败永久挂住进程。
- [x] 将设置和进度写入串行化：同一时刻只允许一个状态写任务；立即保存需要取消并等待旧防抖任务收尾，不能并发复用同一个 `.tmp` 文件。
- [x] 原子写入的临时文件使用单写者保证或唯一临时文件名；完成替换后清理遗留临时文件，并测试进程中断后的恢复行为。
- [x] 增加单实例保护：使用当前用户范围的命名 Mutex；第二次启动不得创建第二套窗口、托盘、热键和 WebView2。
- [x] 第二次启动时向既有实例发送“显示/恢复”请求后立即退出；MVP 使用最小的命名事件或本地 IPC，不引入常驻服务和通用消息总线。
- [ ] 既有实例正在退出时，第二次启动应等待短暂收尾或给出明确提示；禁止两个实例同时写 `%LocalAppData%\AfterglowReader`。
- [ ] `async void` 仅保留框架事件入口；入口内部立即委托给可等待的 `Task` 方法，并统一捕获 I/O、Dispatcher、对话框和 ObjectDisposed 异常。
