你这个项目现在更像一个“固定尺寸的悬浮窗”，而不是一个“可自由调整的桌面应用窗口”。

如果目标主要是 Windows 使用，并且用户需要自由调整窗口大小，那么建议：

去掉或弱化“无边框 + 置顶 + 非激活 + 任务栏隐藏”的默认策略
恢复标准 Windows 窗口行为，至少允许拖拽缩放
把“悬浮/穿透/隐藏”做成可切换模式，而不是默认模式
阅读区域和 WebView2 内容本身要支持响应式布局，避免窗口缩放后排版崩掉
补上最基本的 Windows 桌面 UX：最小化、最大化、恢复、拖动窗口、调整边框、记忆窗口大小和位置
你现在代码里的几个关键点
1. 窗口被设置成了无边框和置顶
在 MainWindow.xaml 里：

WindowStyle="None"
ShowActivated="False"
ShowInTaskbar="False"
Topmost="True"
Opacity="0.94"
这组配置意味着：

没有系统标题栏
没有系统边框
不能自然拖拽缩放
默认不显示任务栏
永远置顶
更像悬浮层，而不是普通应用窗口
这对“阅读浮窗”很合理，但对“窗口大小可以随意调整”不友好，因为 Windows 用户默认会期望：

能拖动窗口边缘缩放
能最大化/最小化
能从任务栏恢复
能通过标题栏操作
2. 还有鼠标穿透和隐藏逻辑
在 MainWindow.xaml.cs 里有：

private bool _clickThrough;
private bool _isHidden;
以及：

else if (message == PlatformNativeWindow.WmNcHitTest && _clickThrough)
{
    handled = true;
    return new IntPtr(PlatformNativeWindow.HtTransparent);
}
这说明项目明显有“阅读时穿透鼠标/隐藏窗口”的设计目标。这个方向没问题，但建议作为“模式”而不是默认状态，否则会让普通 Windows 用户觉得这个窗口“不像正常窗口”。

3. 窗口初始位置是右下角悬浮定位
private void PositionBottomRight()
{
    var workArea = SystemParameters.WorkArea;
    Left = Math.Max(workArea.Left + 12, workArea.Right - Width - 24);
    Top = Math.Max(workArea.Top + 12, workArea.Bottom - Height - 24);
}
这也是典型的浮窗行为，不是标准桌面窗口行为。
如果你要支持自由调整，窗口应该更像“用户自己决定摆哪里”，而不是你强行放在某个角落。

4. WebView2 内容本身是响应式的，但还不够完善
你嵌在 WebView2 里的 HTML 有一些基础布局：

main { display: grid; grid-template-columns: 150px minmax(0, 1fr); gap: 16px; }
这说明内容布局已经具备一定适配性，但只有一个固定的章节栏宽度，窗口缩小时可能会显得拥挤。
所以窗口可缩放之后，Web 内容也要跟着做响应式调整，比如：

小窗口时把目录栏折叠为顶部菜单
章节列表改成抽屉式或可收起
字号、行距根据窗口宽度做轻微自适应
我的建议，按优先级排序
第一优先级：把窗口恢复成“标准 Windows 窗口”
如果你的核心需求是“大小可随意调整”，这一步最重要。

建议改成：

WindowStyle="SingleBorderWindow" 或 WindowStyle="ThreeDBorderWindow"
ResizeMode="CanResize"
ShowInTaskbar="True"
ShowActivated="True"
Topmost="False" 或做成可开关
保留 MinWidth / MinHeight
这样用户才能像普通 Windows 应用一样操作。

如果你还想保留一点“阅读器风格”，可以：

保留圆角、浅色背景、轻量阴影
但不要牺牲窗口基础交互
第二优先级：把“浮窗模式”做成可选模式
你现在其实有两类场景：

场景 A：普通阅读窗口
可拖动
可缩放
可最大化
显示任务栏
不置顶
适合日常使用
场景 B：悬浮阅读窗
置顶
无边框
可穿透
快速隐藏/恢复
适合沉浸式阅读
建议把它们拆开，不要混在默认状态里。

也就是说：

默认启动进入“普通窗口模式”
用户可以切换到“悬浮模式”
悬浮模式再支持 F8 隐藏、穿透、透明度调节
这样更符合 Windows 用户习惯。

第三优先级：实现窗口拖拽缩放体验
如果你坚持用 WindowStyle="None"，那就要自己补完整边框交互，否则用户会觉得卡。

可选方案：

简单方案

在窗口顶部做一个自定义标题栏
标题栏可拖动
右上角加最小化/最大化/关闭按钮
四周留出可缩放热区
中等方案

保持无边框
通过 WM_NCHITTEST 自己实现窗口边缘缩放
这对 Windows 体验更自然
推荐方案

直接用标准窗口框架
只自定义内部视觉
不自己造系统窗口行为
对你这个项目来说，我更推荐第 3 种。因为项目核心是阅读，不是做复杂窗体壳。

第四优先级：记住窗口大小和位置
这个需求对 Windows 桌面应用很重要。

建议在关闭时保存：

Left
Top
Width
Height
WindowState
打开时恢复上次状态。

你现在已经在保存阅读进度和透明度了：

_stateStore.SaveSettingsAsync(new ReaderSettings(Opacity: Opacity)).GetAwaiter().GetResult();
完全可以顺手扩展成保存窗口布局设置。
这会让“可随意调整大小”变成真正可用，而不是每次打开都要重新拖。

第五优先级：让 WebView2 内容真正适配窗口变化
窗口可缩放后，页面内容要跟着变。

建议你在 Web 内容中增加：

@media 响应式布局
小窗时隐藏目录栏
保持正文列宽舒适，例如 max-width
使用 clamp() 控制字体
给正文容器加自适应内边距
比如阅读体验里更合理的原则是：

窗口越宽，正文最大宽度不要无限拉长
窗口越窄，目录栏应收起或变窄
文字行长保持在较舒适范围
这比单纯让窗口变大更重要。

第六优先级：把一些“开发期 P0 验证”功能收敛掉
当前标题还是：

Title="余光阅读器 · P0 WebView2 验证"
这说明它还是验证阶段。
如果你准备长期做 Windows 阅读器，建议尽快把验证性质的设计改成产品化设计，比如：

明确主窗口模式
明确设置窗口
明确阅读窗口
明确悬浮辅助模式
现在这些功能有点混在一起，后面会越来越难维护。

我对这个项目的产品建议
如果它是一个 Windows 阅读器，我建议定位成下面这种结构：

主窗口
标准可缩放窗口
显示书库 / 文件打开 / 最近阅读
任务栏常驻
可最大化、最小化、自由缩放
阅读窗口
也可自由缩放
默认有系统窗口行为
提供阅读模式切换
允许全屏/无边框阅读，但不是默认
悬浮模式
仅当用户主动开启
置顶、穿透、透明度、隐藏快捷键
作为高级功能
这样会比现在“一个窗口承担所有角色”稳定得多。

如果只改最关键的一件事
如果你现在只准备做一个最小改动，我建议优先改 MainWindow.xaml：

去掉 WindowStyle="None"
去掉 ShowInTaskbar="False"
去掉 ShowActivated="False"
去掉默认 Topmost="True"
然后保留：

MinHeight
MinWidth
现有的圆角视觉
阅读内容 WebView2
这是最直接、最能让 Windows 用户感到“这是个正常窗口”的方式。



我建议的架构分层
1. UI 层
只负责：

显示窗口
响应按钮点击
展示状态
把用户动作转发给服务层
这一层不应该直接干：

读文件
写持久化
调 WebView2 注入脚本
注册热键
操作系统 API
现在你的 MainWindow.xaml.cs 已经在做这些事了，所以它未来会越来越难维护。

2. 应用协调层
建议新增一个“应用控制器”或“阅读会话控制器”。

它负责：

打开书籍
切换当前书籍
管理阅读进度
管理窗口模式
处理中断和恢复
把 UI 事件串起来
这层相当于“业务编排中心”，可以避免 MainWindow 变成一个巨型类。

3. 核心阅读层
负责：

文本解析
EPUB/MOBI/TXT 处理
章节生成
段落结构化
阅读进度计算
搜索、目录、分页等阅读逻辑
这个层应该尽量独立于 WPF 和 WebView2。
这样做的好处是：

UI 换成别的框架也不用重写核心逻辑
单元测试更容易
出错范围更小
4. 基础设施层
负责：

文件访问
本地设置保存
最近阅读记录
日志
依赖系统 API 的功能，比如托盘、热键、窗口样式控制
这层最好做成接口化，比如：

ISettingsStore
IProgressStore
IHotkeyService
INotificationService
IWindowNativeService
这样未来如果你要替换实现，不会动到核心业务。

重点问题和稳定性建议
1. 不要让 MainWindow 直接做业务编排
你现在的 MainWindow.xaml.cs 做了很多事：

启动时初始化状态
绑定托盘
注册热键
初始化 WebView2
打开文件
渲染书籍
保存进度
保存设置
处理鼠标穿透
处理隐藏恢复
这很容易出问题，因为任何一个环节异常，都会影响整个窗口。

建议
把这些逻辑拆出去：

MainWindow 只保留 UI 事件
ReaderSessionService 负责当前阅读状态
WindowBehaviorService 负责隐藏、穿透、置顶、显示
BookLoadingService 负责打开和解析书籍
ReaderPersistenceService 负责保存和恢复
这样每个模块都能单独出问题、单独修。

2. WebView2 要做成“可重建”的渲染目标
WebView2 是项目里最容易出现稳定性问题的地方之一。
原因是它有自己的进程、自己的初始化时机、自己的页面状态。

风险
初始化失败
导航失败
JavaScript 注入失败
进程崩溃
页面状态丢失
建议
不要把 WebView2 当成“永远存在且永远正常”的控件。
应该把它当成“可重建渲染器”。

也就是说，架构上要支持：

WebView2 初始化失败后重试
WebView2 进程崩掉后重建
当前书籍状态可重新渲染
页面内容来源于外部状态，不依赖页面内不可恢复状态
这点非常关键。

最佳实践
把当前书籍内容保存在一个独立对象里，例如：

当前书籍模型
当前滚动位置
当前章节
当前设置
然后 WebView2 只是“展示层”，坏了可以重新初始化后重新渲染。

3. 所有异步操作都要有取消和兜底
现在你很多方法是 async void，比如：

OnLoaded
OpenFilePlaceholder
这在桌面应用里能用，但稳定性一般不够。

问题
异常不容易统一处理
任务无法取消
窗口关闭时可能还有后台任务跑着
多次点击可能导致并发冲突
建议
UI 事件可以是 async void
真正的业务逻辑尽量是 async Task
加一个全局 CancellationTokenSource
窗口关闭时取消所有未完成任务
例如：

打开书籍时如果用户又点了另一份文件，前一个任务应取消
窗口关闭时，WebView2 初始化、文件解析、保存任务都应尽量停止
4. 保存状态和读取状态要“原子化、容错化”
你现在已经有进度存储和设置存储，这是好事，但稳定运行还要考虑：

文件损坏
写入中断
版本升级后格式变化
多次保存冲突
建议
持久化层做成：

写入时先写临时文件
再原子替换正式文件
读取失败时用默认值兜底
对配置版本加字段版本号
这样可以避免应用因为配置坏掉而启动失败。

5. 把系统 API 封装起来
你现在的 NativeWindow 里已经放了一些 Windows API：

热键注册
点击穿透
无激活显示
扩展样式修改
这个方向是对的，但建议继续向“封装服务”升级，不要让业务层直接碰 P/Invoke。

推荐结构
NativeWindow 保留底层 P/Invoke
上层提供 WindowBehaviorService
UI 或控制器只调用这个服务
好处：

更好测试
更好排错
更不容易把 Win32 细节散落到各处
6. 书籍解析与渲染分离
现在你的流程大概是：

选文件
BookLoader.LoadAsync
JsonSerializer.Serialize(book)
ExecuteScriptAsync
这条链路是可用的，但长期稳定运行最好再拆一层：

逻辑建议
BookLoader 负责读取和解析
BookDocumentTransformer 负责把领域模型转成适合渲染的数据
ReaderRenderer 负责把数据送进 WebView2
这样未来你如果要支持：

分页模式
连续滚动模式
夜间模式
排版参数
字体设置
就不会把解析层改乱。

7. 全局错误处理要补齐
稳定运行必须有统一异常出口。

建议至少有三层保护
a. UI 事件层
捕获按钮点击、窗口事件里的异常，防止直接崩溃。

b. 任务层
所有 Task 里的异常要统一记录，必要时给用户提示。

c. 全局未处理异常
建议在应用启动时注册：

UI 线程未处理异常
Task 未观察异常
AppDomain 未处理异常
然后做：

记录日志
弹出友好提示
尝试保存当前状态
避免直接静默退出
8. 加日志系统，这是稳定运行的核心
如果你想要“保证稳定运行”，日志几乎是必须的。

日志应该记录
应用启动流程
WebView2 初始化结果
文件打开结果
解析异常
渲染异常
热键注册失败
窗口显示/隐藏状态切换
保存状态结果
建议
用结构化日志，而不是简单 Console.WriteLine。
桌面应用很推荐：

Serilog
NLog
至少要做到：

本地滚动日志
崩溃后可回溯
分级记录 Info / Warning / Error
9. 模式状态要集中管理，不要散在多个 bool 里
你现在有：

_isHidden
_clickThrough
_bossHotKeyRegistered
未来功能一多，这些状态会越来越散。

建议
建立一个统一的 ReaderWindowState 或 AppState，比如：

当前书籍
是否隐藏
是否穿透
是否置顶
窗口位置大小
透明度
当前阅读模式
当前章节
统一保存、统一恢复、统一变更。

这样可以避免“某个布尔值变了但其他地方没同步”的问题。

10. 主线程和后台线程边界要清晰
WPF + WebView2 特别要注意线程问题。

原则
UI 只能在 UI 线程改
文件解析放后台
渲染回到 UI 线程
状态保存尽量后台化，但关闭时要兜底等待
建议
如果后面功能增多，最好引入一个明确的“调度策略”：

UI 操作回主线程
长耗时解析进后台
最终渲染回 UI
这能显著降低卡顿和死锁风险。

如果你要一个“稳定优先”的推荐架构
我建议你采用这个简单可执行的结构：

App
负责：

启动
全局异常处理
依赖注入组装
应用生命周期
MainWindow
负责：

展示
接收交互
调用控制器
ReaderController
负责：

打开书籍
切换书籍
控制渲染
协调保存恢复
ReaderRenderer
负责：

WebView2 初始化
页面加载
渲染书籍
重建 WebView2
BookLoader
负责：

TXT/EPUB/MOBI 解析
构建统一文档模型
AppStateStore
负责：

设置保存
进度保存
窗口布局保存
WindowBehaviorService
负责：

置顶
隐藏
穿透
非激活显示
Win32 API 封装
LoggingService
负责：

错误记录
运行日志
对你当前项目的现实建议
如果你不想一下重构太大，我建议按这个顺序做：
